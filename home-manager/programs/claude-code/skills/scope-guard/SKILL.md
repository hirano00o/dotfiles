# 変更スコープ管理

## 基本方針

- **Touch only what is necessary**: タスクに直接関係しないコード・ファイルに手を入れない
- **最小差分**: 行数・ファイル数を最小に保つ。冗長な追加は後から削除される
- **根本原因を修正**: 症状を隠す一時対応 (workaround) ではなく、根本原因に手を入れる
- **ハックに感じたら立ち止まる**: 回避策で進めそうになったら、他に取りうる最善の方法がないか再検討

## ドキュメント同期

コード変更時、以下を**同一コミット**で更新する (CLAUDE.md の MUST):

- 公開 API の docstring / 型注釈
- README / 設計ドキュメント / ADR
- CHANGELOG (破壊的変更・新機能追加時)
- 設定・環境変数の例ファイル (`.env.example` 等)

## 境界線

実装中に以下を発見した場合の扱い:

- **無関係のバグ**: タスクを記録 (TodoWrite / メモ) し、現在のサイクル完了後に別タスクで対応
- **無関係のリファクタ機会**: 同上。混ぜ込まない
- **依存関係の更新**: 現在のタスクに必要なら行う。そうでなければ別タスク
- **型・テストの既存エラー**: 現タスクの範囲で解消できるなら対応。大規模な場合は別タスク化

## コミット粒度

- 1 コミット = 1 意味ある変更単位
- 実装 + テスト + ドキュメントは同一コミット
- リファクタはコミットを分ける (実装コミットと混ぜない)
- 破壊的な取り消しを避けるため、巨大コミットを作らない

## 確認

コミット作成前:

- `git diff --stat` で影響範囲を確認し、想定外のファイルが含まれていないか
- 意図しない変更 (インデント修正、エディタによる自動整形など) が混入していないか
- デバッグ用の一時コード (console.log, print, fmt.Println) が残っていないか
