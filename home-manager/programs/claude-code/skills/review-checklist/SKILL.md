# レビュー視点チェックリスト

実装完了を宣言する前に、以下を自問する。**"Would a staff engineer approve this?"**

## 1. 正確性

- 仕様に書かれた振る舞いを全て満たしているか
- エッジケース (境界値、空入力、null / undefined、エラー経路) を扱えているか
- Off-by-one、型変換、並行アクセスの取り違えがないか
- 失敗経路のロールバック・クリーンアップが適切か

## 2. セキュリティ

- ユーザー入力の検証・サニタイズが行われているか (SQL インジェクション / XSS / パストラバーサル)
- シークレット・API キーがソースおよびコミットに混入していないか
- 権限チェック・認証が期待通りに機能しているか
- 信頼できないデシリアライズや外部 URL 解決が行われていないか

## 3. パフォーマンス

- N+1 クエリや不要なループ反復がないか
- 巨大オブジェクトのコピー・不要な文字列連結が発生していないか
- Goroutine / async タスクのリークがないか (キャンセル伝播が正しいか)
- 同期処理がクリティカルパスを塞いでいないか

## 4. 保守性

- 命名が意図を表現しているか (how を説明する識別子)
- 関数・モジュールが単一責任になっているか
- 過度な抽象化・早すぎる一般化をしていないか (3 回繰り返してから共通化)
- コメントは "why not" (非自明な決定) のみに絞られているか

## 5. テスト十分性

- 単機能ごとにテストが存在するか
- 失敗経路・エラーハンドリングもテストされているか
- テストが実装詳細ではなく振る舞いを検証しているか
- モックは境界 (外部 API など) にのみ使われ、内部ロジックはモックしていないか

## 6. ドキュメント

- 公開インターフェースに docstring / コメントがあるか
- 関連する README / 設計ドキュメントを同一コミットで更新しているか
- 破壊的変更がある場合、CHANGELOG / マイグレーション手順があるか
