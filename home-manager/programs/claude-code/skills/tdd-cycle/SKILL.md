# TDD サイクル

## 原則

- **テストは "what"**: 期待される振る舞いを表現する
- **実装は "how"**: その振る舞いをどう実現するかを書く
- **コミットメッセージは "why"**: なぜその変更を行うかを残す
- 1 サイクル = 1 単機能 (最小粒度)。仕様を分解してから着手する

## サイクル

1. **仕様分解**: 受け取った仕様を独立した単機能に分ける。依存順に並べ、最初に着手するものを 1 つ選ぶ
2. **Red**: 着手対象の振る舞いを検証する失敗するテストを 1 つ書く
3. **実行**: テストが "期待通りに失敗" することを確認する (実装未着手でテストが通ったらテストが誤っている)
4. **Green**: 最小限のコードでテストを緑にする。この段階では最適化・抽象化しない
5. **Refactor**: 緑を維持したままコードを整理する。重複除去、命名改善、意図の明確化
6. **Lint**: 言語固有のリンタを実行。失敗したら Refactor に戻る
7. **Format**: 言語固有のフォーマッタを適用
8. **次サイクル**: 全テストが緑なら、次の単機能へ

## ガード

- サイクル途中で別機能の実装・リファクタに寄り道しない
- 1 サイクル完了前にコミットを作らない (中途半端な状態を残さない)
- テストが緑にならないまま他の変更に進まない
- 全サイクル完了後、全体を通しで実行してリグレッションがないことを確認する
